## 0. 🔴 系统初始化与动态上下文加载 (最高优先级)

为了保证开发上下文的连贯性与准确性，在每次开启新任务或会话时，必须**动态加载**以下两个核心状态文件。**严禁在未读取当前项目上下文的情况下盲目编码。**

1. **`Progress.md` (项目上下文与进度追踪)**

2. **`Memory.md` (经验教训与全局约束)**
   - **作用**：存放项目开发过程中沉淀的经验教训、踩过的坑、特定环境的配置要求以及约定的最佳实践。
   - **行动**：遇到问题规划技术方案前，必须检索此文件以避免重复犯错；解决复杂 Bug 后，必须将复盘结果写入此文件。

## 1. 核心原则

### 1.1 需求优先

- **不盲目编码**：充分理解需求和业务场景后再行动。遇到边界模糊或需求不明确时，主动向用户提问确认。
- **可追溯性**：所有的代码变更必须能对应到具体的业务需求或技术优化目标。

### 1.2 质量至上

- **流程不可省**：严格遵循“设计-编码-测试-审查”的标准流程，不跳过任何关键步骤。
- **测试前置**：编写核心逻辑的同时必须配套对应的测试用例。
- **代码整洁**：高度重视代码的可读性、可维护性和高内聚低耦合。

### 1.3 持续迭代与安全防护

- **小步快跑**：将大任务拆分为小粒度的提交，快速验证，及时反馈。
- **安全可靠**：确保代码可回滚，不破坏现有功能；处理敏感数据或执行破坏性操作（如删除文件、修改数据库）前必须多重确认。

## 2. 标准开发流程

在本项目中，必须严格执行以下工作流。任何跳过测试直接交付的行为都是被禁止的。

```text
1. 需求对齐 (读取 Progress.md) → 2. 方案设计 (参考 Memory.md) → 3. 代码实现 → 4. 测试验证 → 5. 更新上下文 (回写 Progress.md/Memory.md) → 6. 部署交付
```

### 关键阶段输出物要求：

- 方案设计：明确技术选型、接口定义、潜在风险及规避策略。
- 代码实现：符合所用语言的标准规范，添加必要的业务逻辑注释。
- 测试验证：涵盖正常路径和异常路径（Happy/Sad paths）。
- 更新上下文：文档必须与代码保持绝对同步。

## 3. 技术与代码规范

### 3.1 代码级规范

- Python：严格遵循 PEP 8 规范，强制使用类型注解（Type Hints）。使用 uv 管理依赖。
- Node.js/Web：强制开启 TypeScript 严格模式，遵循 ESLint 规范。使用 npm 管理依赖。
- 职责单一：函数与类的设计需遵循单一职责原则（SRP），控制圈复杂度。

### 3.2 日志规范

在并发/异步或多进程环境中，日志必须包含足够的上下文：

- 必填字段：`[时间戳] [日志级别] [线程/进程ID] [模块名] - 业务标识: 详细信息`

异常捕获必须打印完整的堆栈信息（Stack Trace）。

### 3.3 版本控制

- 提交（Commit）粒度要小且独立，遵循 Conventional Commits 规范（如 `feat:`, `fix:`, `docs:`, `refactor:`）。
- 绝不提交包含任何敏感信息（密钥、Token、内部 IP）的文件。

## 4. 质量保证体系 (QA)

### 4.1 代码审查清单

执行自检或交付前，逐一核对以下指标：

- [ ] 逻辑正确性：是否满足 `Progress.md` 中的验收标准？
- [ ] 边界条件：是否处理了空值、越界、超时等异常情况？
- [ ] 性能考量：是否存在 N+1 查询、死循环隐患或 O(n²) 以上的低效算法？
- [ ] 安全审查：是否存在硬编码密钥、SQL 注入、跨站脚本（XSS）隐患？

### 4.2 测试金字塔策略

- 单元测试 (Unit Tests)：覆盖所有核心业务逻辑与工具函数，要求覆盖率 > 80%。单次执行时间控制在毫秒级。
  - 强制 Mock：必须 Mock 所有外部依赖（数据库、网络请求、文件系统），确保测试环境纯洁且无副作用。

- 边界优先：除核心路径外，必须强制验证异常边界条件（Edge Cases），如空值、极大值、格式错误等。
  - 集成测试 (Integration Tests)：验证模块间的协作、数据库交互和外部 API 调用。

- 状态隔离：必须包含严格的数据清理逻辑（Setup & Teardown），确保测试用例之间相互独立，防止脏数据污染。
  - 端到端测试 (E2E 测试)：验证关键用户链路。

### 4.3 E2E 测试排障与调试指南 (AI 与开发者协作专属)
E2E 测试环境复杂，当自动化测试失败时，模型或开发者排查问题必须严格遵循以下标准工作流，避免盲目猜测：

1. **精准定位失败节点**：
首先提取测试框架（如 Playwright/Cypress/Selenium）报错的精确行号、失败的断言（Assertion）以及超时提示。

2. **收集并分析上下文资产 (Artifacts)**：
   - DOM 快照/截图：检查 UI 渲染状态，确认目标元素是否被遮挡、未渲染或不在视口内。
   - 网络请求 (Network)：检查失败节点前后的 API 状态码（是否 4xx/5xx）、请求体（Payload）和响应数据。
   - 控制台日志 (Console)：排查是否有未捕获的 JavaScript 运行时报错导致页面崩溃。

3. **状态与时序排查 (最常见雷区)**：
   - 数据预置问题：确认测试执行前的数据库或应用状态是否被正确 Seed（植入），是否存在前置测试遗留的脏数据。

   - 异步等待问题：分析失败是否由于网络延迟或前端动画未结束导致。严禁使用硬编码的延时（如 sleep(5000)），必须使用基于状态的断言等待（例如：等待元素变为可见、等待特定网络请求返回 HTTP 200）。

4. **构建最小复现步骤**：

   - 提取导致失败的单个测试用例，在 PROGRESS.md 中记录独立的复现命令（如 npx playwright test --grep "login user flow" --debug），逐步实施修复验证。

## 5. 知识库与状态更新规范

### 5.1 经验教训沉淀 (Memory.md)

遇到复杂 Bug 或完成重要架构重构后，必须按以下格式向 Memory.md 追加记录，确保“同样的坑不踩两次”：

```markdown
### [日期] 问题/经验简述
- **问题描述**：[清晰描述现象、触发条件及错误信息]
- **根本原因**：[深入分析导致问题的原因]
- **解决方案**：[详细的修复步骤或重构方案]
- **预防措施**：[转化为开发规范、Lint规则或测试用例，防止再次发生]
```

### 5.2 项目状态推进 (Progress.md)

每次任务完成时，必须更新 Progress.md 的以下部分：

- 将已完成的任务从 TODO 移至 DONE。
- 简要记录本次更新的核心系统变更（新增了什么模块、修改了什么接口）。
- 补充下一步的前置依赖或新发现的待办事项。

## 6. 安全与灾备

### 6.1 敏感信息保护

使用 .env 文件存储敏感配置，并确保 .env 在 .gitignore 中。

严禁在代码注释、日志、或非加密文档中记录真实的密码或生产环境 Token。

### 6.2 数据防丢与恢复

操作前备份：在执行大批量数据修改、文件覆盖、强制 Git 操作（如 reset --hard）前，必须确认已进行本地或远程备份。

若发生意外丢失，优先从 .bak 备份或 Git 历史中恢复，并将事故记录至 Memory.md 中。

行动宗旨：始终保持上下文清晰，先思考后编码，用文档固化认知，用测试捍卫质量。
